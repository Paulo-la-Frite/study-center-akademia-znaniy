# Анализ логической модели данных (Часть I)

## Исходная модель данных

Исходная логическая модель из курсовой работы включала 7 таблиц:
- Groups (Группы)
- Students (Студенты)
- Teachers (Преподаватели)
- Subjects (Дисциплины)
- Schedule (Расписание)
- Grades (Успеваемость)
- Attendance (Посещаемость)

## Результаты анализа (статическое тестирование)

В ходе анализа исходной модели были выявлены следующие недостатки:

1. **Отсутствие единой таблицы пользователей.** Учётные данные для аутентификации были распределены между таблицами Students и Teachers. Роли "Методист" и "Администратор" не имели механизма аутентификации. Это противоречит требованиям части V задания, где требуется единая таблица "Пользователи".

2. **Отсутствие полей для требований безопасности.** Требования части V предписывают:
   - Блокировку учётной записи после 3 неверных попыток входа
   - Блокировку при неактивности более 1 месяца
   - Принудительную смену пароля при первом входе
   Исходная модель не содержала полей для реализации этих функций.

3. **Именование полей.** Часть полей использовала смешанный стиль (русские названия, CamelCase), что не соответствует рекомендуемому стилю snake_case для PostgreSQL.

## Внесённые изменения

1. **Создана таблица users** — центральная точка аутентификации для всех ролей.
2. **Добавлены связи** — внешний ключ user_id в таблицах students и teachers.
3. **Добавлены поля безопасности:**
   - is_blocked (BOOLEAN) — признак блокировки
   - failed_login_attempts (SMALLINT) — счётчик неверных попыток
   - password_must_change (BOOLEAN) — требование смены пароля
   - last_login (TIMESTAMP) — дата последнего входа
4. **Все имена приведены к snake_case**.
5. **Определены ограничения:** CHECK, UNIQUE, NOT NULL, DEFAULT для всех полей.

## Нормализация (3НФ)

Модель проверена на соответствие трём нормальным формам:

- **1НФ:** Все атрибуты атомарны, повторяющиеся группы отсутствуют.
- **2НФ:** Каждый неключевой атрибут зависит от полного первичного ключа. Частичные зависимости устранены (справочники вынесены в отдельные таблицы).
- **3НФ:** Транзитивные зависимости отсутствуют. Например, данные о группе не хранятся в таблице студентов, а вынесены в справочник groups.

## Итоговая модель (8 таблиц)

1. users — пользователи системы
2. groups — учебные группы
3. students — студенты (связь с users)
4. teachers — преподаватели (связь с users)
5. subjects — дисциплины
6. schedule — расписание занятий
7. grades — успеваемость
8. attendance — посещаемость
