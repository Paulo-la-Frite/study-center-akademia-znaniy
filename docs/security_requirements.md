# Требования безопасности ИС (Часть IV)

## 1. Парольная политика
- Пароли хранятся в базе данных исключительно в хэшированном виде (алгоритм bcrypt, 12 раундов).
- Минимальная длина пароля — 6 символов.
- При первой авторизации с временным паролем, выданным администратором, пользователь обязан сменить пароль.
- Новый пароль должен совпадать с подтверждением.

## 2. Защита от перебора учётных данных
- После трёх неверных попыток входа учётная запись блокируется.
- Счётчик неудачных попыток (failed_login_attempts) увеличивается при каждом неверном вводе.
- Счётчик сбрасывается при успешной авторизации.
- При блокировке выводится сообщение: "Вы заблокированы. Обратитесь к администратору."
- Разблокировка возможна только через панель администратора.

## 3. Блокировка неактивных учётных записей
- Если пользователь не авторизовался более 1 месяца, учётная запись автоматически блокируется.
- Проверка выполняется при каждой попытке аутентификации (поле last_login).

## 4. Разграничение доступа (авторизация)
- Система использует ролевую модель: Администратор, Методист, Преподаватель, Студент.
- После успешной аутентификации интерфейс формируется динамически в зависимости от роли.
- Пользователь видит только те функции, которые разрешены его роли.

## 5. Защита от SQL-инъекций
- Все SQL-запросы выполняются через параметризованные запросы (psycopg2, плейсхолдеры %s).
- Не используется конкатенация строк и форматирование для подстановки пользовательского ввода.

## 6. Защита персональных данных
- Персональные данные студентов и преподавателей хранятся в защищённой БД.
- Доступ к таблицам ограничен на уровне приложения в соответствии с ролью.
- Пароли не логируются и не отображаются в открытом виде нигде в системе.

## 7. Логирование
- Фиксируются все попытки входа (успешные и неуспешные).
- Время последнего входа записывается в поле last_login.
